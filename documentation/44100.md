# AppleFWAudio 44.1 kHz Family: Reverse-Engineering Notes

**Status:** active research note. This document records observations from the
AppleFWAudio Mach-O loaded in IDA, not an implementation specification.

**Evidence source:** IDA function names, disassembly, cross-references, and
embedded diagnostics from the AppleFWAudio binary. The local `references/`
directory was absent when this work began. Later comparison sections use the
user-supplied, read-only Linux and FFADO trees; the Apple binary findings
remain unverified against an Apple source release. Do not copy binary-derived,
GPL, or LGPL source code; use behavioral observations only.

> [!IMPORTANT]
> **ASFW implementation boundary:** ASFW does not use Apple's DCL or NuDCL
> architecture. It is a DriverKit reimplementation of the FireWire/OHCI stack
> for macOS Tahoe, with its own descriptor, DMA, lifecycle, and packet-framing
> layers. DCL and NuDCL are historical reference mechanisms only. This note
> extracts candidate wire-level invariants from them; it does not prescribe
> either mechanism, layout, callback model, or memory ownership for ASFW.

---

## 1. Findings at a glance

AppleFWAudio contains two distinct transmit representations for the 44.1 kHz
family:

1. **Legacy `AM824DCLWrite`** builds a static DCL program with one isochronous
   packet per 8 kHz FireWire cycle. It uses an integer Bresenham/DDA accumulator
   to select either of two adjacent frame counts per packet:
   - 44.1 kHz: 5 or 6 frames
   - 88.2 kHz: 11 or 12 frames
   - 176.4 kHz: 22 or 23 frames
2. **Later `AM824NuDCLWrite`** uses fixed SYT intervals (8, 16, or 32 frames)
   and schedules data versus no-data cycles from a phase accumulator. A
   no-data cycle writes `SYT = 0xffff` and has no payload.

Both representations deliver the same average sample rate, but they are not
wire-identical packet schedules. That disagreement is evidence to resolve, not
an implementation choice for ASFW: any Tahoe packetizer behavior must be
cross-validated against captures and read-only reference stacks before it is
made wire-visible.

## 3. Legacy rate dispatch

`AM824DCLWrite::SetupSendBuffer()` reads the configured sample rate and
dispatches as follows:

| Sample rate | Argument passed to `SetupSendBufferForMultipleOf44100Hz` |
|---:|---:|
| 44,100 | `441` |
| 88,200 | `882` |
| 176,400 | `1764` |

IDA evidence: `0x3681f`, `0x3682b`, and `0x36837`, respectively. The argument
is therefore **sample rate divided by 100**, not a 1x/2x/4x multiplier. The
decompiler's `sampleRateMultiplier` name is misleading.

The 8 kHz-multiple path handles 32,000, 48,000, 96,000, and 192,000 Hz. It
calculates `framesPerPacket = sampleRate / 8000`, which is exact for those
rates: 4, 6, 12, and 24 frames per FireWire cycle.

## 4. The 44.1-family accumulator

The function begins at `0x3701c` by calculating:

```text
base      = rateDiv100 / 80
remainder = rateDiv100 % 80
```

The apparent multiply by `0xcccccccd` and shift at `0x3701d..0x3703f` is the
compiler's unsigned division-by-80 sequence. `80` is the scaled form of the
8,000 FireWire cycles/second denominator:

```text
(sampleRate / 100) / (8000 / 100) = sampleRate / 8000
```

The accumulator is initialized to zero at `0x371f4`. Each packet-DCL built by
the inner loop at `0x3726c..0x372f5` follows this behavior:

```cpp
uint32_t error = 0;

for (each_8kHz_isoch_cycle) {
    error += remainder;
    uint32_t frames = base;

    if (error >= 80) {
        error -= 80;
        ++frames;
    }

    build_packet_dcl(frames);
}
```

This is Bresenham-style error accumulation, also called a digital differential
analyzer (DDA). It distributes the fractional part of `sampleRate / 8000`
without floating point or long-term rate error.

The code uses `error > 79` at `0x37272`, which is equivalent to `error >= 80`.
Because `remainder < 80`, at most one carry is required per packet.

### Bresenham/DDA mechanics

The accumulator is the same algorithm used to draw a line whose slope is not
an integer: advance one unit along the horizontal axis every step, retain the
fractional vertical distance, and take an extra vertical step only when the
retained fraction reaches one whole unit.

For audio, a horizontal step is one 125 us FireWire cycle. The vertical steps
are audio frames. In unscaled form:

```text
sampleRate = base * 8000 + remainder
```

After `n` packet decisions from a zero seed, the legacy algorithm guarantees:

```text
error(n)  = (n * remainder) mod 8000
frames(n) = n * base + floor(n * remainder / 8000)
          = floor(n * sampleRate / 8000)
```

The error is always in `[0, 7999]`. Thus it cannot grow into clock drift: it
is only the still-unemitted fractional part of a frame. Apple divides both
terms by 100 for the 44.1-family builder, so the binary uses the equivalent
`80` denominator and `41` remainder for 44.1 kHz.

Worked 44.1 kHz example using Apple's scaled values:

| Cycle | Error before carry | Frames in this packet | Error retained |
|---:|---:|---:|---:|
| 1 | `0 + 41 = 41` | 5 | 41 |
| 2 | `41 + 41 = 82` | 6 | 2 |
| 3 | `2 + 41 = 43` | 5 | 43 |
| 4 | `43 + 41 = 84` | 6 | 4 |
| 5 | `4 + 41 = 45` | 5 | 45 |
| 6 | `45 + 41 = 86` | 6 | 6 |

The packet with 6 frames is not a correction after an error. It is the packet
that carries the one accumulated fractional frame. Over 80 cycles, the 41
carries become the 41 six-frame packets required for exactly 441 frames.

### Blocking form of the same rate equation

NuDCL changes the unit from one audio frame to one fixed 8-frame event. At
44.1 kHz, the required data-event rate is:

```text
(44100 frames/second) / (8 frames/event) / (8000 cycles/second)
= 441 data events / 640 cycles
```

A payload-only DDA for that ratio is:

```cpp
eventError += 441;
if (eventError >= 640) {
    eventError -= 640;
    emitDataPacket(8);
} else {
    emitNoDataPacket();
}
```

From a zero seed it begins `N, D, D, N, ...`; after `n` cycles it has emitted
`floor(n * 441 / 640)` data packets. This proves the same long-term rate as
the legacy 5/6-frame schedule, but packets are intentionally burstier because
each data decision transfers all eight frames at once.

Apple NuDCL does not literally use this short `eventError` form. It stores the
same fractional relation as two FireWire-time deadlines: `packetPhase` advances
once per bus cycle and `nextSytPhase` advances once per 8-frame interval. The
fixed-point phase representation adds the information a simple event counter
lacks: exact SYT presentation time, 16-cycle wrap handling, a presentation
lead, and a resynchronization point.

### Reference pseudocode

The two Apple representations reduce to the following implementation-neutral
pseudocode. These are behavioral models, not source to copy.

```cpp
// Legacy AM824DCLWrite: one variable-size packet every bus cycle.
struct NonBlockingDda {
    uint32_t baseFrames;  // rate / 8000
    uint32_t remainder;   // rate % 8000
    uint32_t error;       // initially 0, always < 8000
};

uint32_t NextNonBlockingFrames(NonBlockingDda& s) {
    s.error += s.remainder;
    uint32_t frames = s.baseFrames;
    if (s.error >= 8000) {
        s.error -= 8000;
        ++frames;
    }
    return frames;
}
```

```cpp
// AM824NuDCLWrite: one fixed-size SYT event when its bus-time deadline is due.
struct BlockingSytPhase {
    uint64_t packetPhase;       // starts at 0
    uint64_t nextSytPhase;      // Apple 44.1 startup seed: 117'220'000
    uint32_t sytIntervalFrames; // 8 at 44.1 kHz
};

constexpr uint64_t kCyclePhase = 30'720'000; // 3072 FireWire ticks * 10000
constexpr uint64_t kPhaseModulo = 491'520'000; // 16 cycles
constexpr uint64_t kHalfRing = kPhaseModulo / 2;
constexpr uint64_t kSytPhaseStep44100 = 44'582'312;
constexpr uint64_t kPresentationLead = 460'800'000;

bool HasReachedDeadline(uint64_t packetPhase, uint64_t deadline) {
    const uint64_t lag = (packetPhase + kPhaseModulo - deadline) % kPhaseModulo;
    return lag != 0 && lag <= kHalfRing;
}

PacketDecision NextBlockingPacket(BlockingSytPhase& s) {
    s.packetPhase = (s.packetPhase + kCyclePhase) % kPhaseModulo;

    if (!HasReachedDeadline(s.packetPhase, s.nextSytPhase))
        return PacketDecision::NoData; // SYT = 0xffff, DBC unchanged

    const uint64_t presentationPhase =
        (s.nextSytPhase + kPresentationLead) % kPhaseModulo;
    const uint64_t ticks = presentationPhase / 10000;

    PacketDecision result{};
    result.frames = s.sytIntervalFrames;
    result.sytCycle = ticks / 3072;
    result.sytOffset = ticks % 3072;

    s.nextSytPhase =
        (s.nextSytPhase + kSytPhaseStep44100) % kPhaseModulo;
    return result;
}
```

The second model deliberately omits the Apple resynchronization branch. That
branch adjusts phase from live bus state and must be recovered separately before
ASFW implements it.

### Exact schedules from the zero seed

| Rate | `base` | `remainder` | Period | Packet-frame schedule | Frames per period |
|---:|---:|---:|---:|---|---:|
| 44.1 kHz | 5 | 41 | 80 cycles | `(5, 6)` repeated 39 times, then `6, 6` | 441 |
| 88.2 kHz | 11 | 2 | 40 cycles | 39 x `11`, then `12` | 441 |
| 176.4 kHz | 22 | 4 | 20 cycles | 19 x `22`, then `23` | 441 |

For 44.1 kHz, the 80-cycle (10 ms) period contains 39 five-frame packets and
41 six-frame packets: `39 * 5 + 41 * 6 = 441`. For 88.2 and 176.4 kHz, the
shorter sequence repeats enough times to make 80 FireWire cycles carry 882 and
1764 frames, respectively.

The accumulator is outside both the packet and buffer-group loops. It is not
reset at a buffer-group boundary. This preserves the distribution through the
entire DCL ring that is constructed in this invocation.

## 5. What the accumulator controls

The accumulator determines the transfer length of each per-packet DCL command.
At `0x37283..0x372a7`, the binary calculates:

```text
quadletCount = 2 + frames * payloadQuadletsPerFrame
transferBytes = 4 * quadletCount
```

`2` is consistent with the two CIP header quadlets. The binary expression for
`payloadQuadletsPerFrame` is the sum of fields at object offset `0x54` and the
per-stream parameter block offset `0x0c`; their source-level names have not
yet been recovered. Its behavior is consistent with the number of audio
quadlets contributed by a frame.

The command's source address advances by the fixed transmit-buffer slot stride,
not by the variable `transferBytes` value. This lets every packet have a
maximum-sized storage slot while the DCL transfer length selects how much of
that slot goes out on the wire.

The function builds the whole DCL program ahead of time:

```text
program start
  -> packet DCLs for every packet in a buffer group
  -> timestamp/update/callback commands for that group
  -> next buffer group
  -> branch back to program start
```

The final branch is installed at `0x37401..0x37413`. For seamless repetition,
the total packet count in the loop must preserve the accumulator phase. The
next investigation should verify that the allocated number of groups times
packets per group is a multiple of the relevant period (80, 40, or 20).

## 6. Buffer-group lifecycle and DCL reuse

`DVAllocatePlayBufferGroup()` at `0x35e36` allocates an 84-byte, 16-byte
aligned `DVLocalOutStruct`, zeroes it, stores its owning global-output pointer,
and inserts it into a circular doubly linked list. It is used by every legacy
send-buffer construction path, including the 44.1-family builder.

The fields established by `SetupSendBufferForMultipleOf44100Hz()` and later
uses are currently understood as follows:

| Offset | Evidence-based role |
|---:|---|
| `0x00`, `0x04` | Next and previous pointers in the circular buffer-group list. |
| `0x08` | Owning global-output structure. |
| `0x0c` | Callback DCL pointer. |
| `0x10` | Timestamp DCL pointer. |
| `0x14`, `0x18` | First and last packet DCL in this group. |
| `0x1c` | Single-DCL update-list target, initialized to the timestamp DCL. |
| `0x20`, `0x24` | Pointer to the packet-DCL update list and its element count. |
| `0x28` | Buffer-group index. |
| `0x38` | 64-bit `1,000,000,000 / sampleRate` value. |

`DVCreatePlayBufferGroupUpdateList()` at `0x35ea0` walks the inclusive
`[firstPacketDCL, lastPacketDCL]` range. It retains only DCLs whose 16-bit type
field at DCL offset `0x08` is `1` or `3`, allocates a 16-byte-aligned pointer
array, and stores that array and count at group offsets `0x20` and `0x24`.

The 44.1-family setup then inserts two update-list DCLs around the callback:

```text
packet DCL chain
  -> timestamp DCL
  -> update list: [group + 0x1c], count 1
  -> callback DCL, argument = group
  -> update list: group[0x20], count group[0x24]
```

The exact hardware semantics of the legacy DCL update-list operation, including
the meaning of DCL types `1` and `3`, remain unresolved. The data flow above is
directly observed; do not infer more from the provisional opcode names.

### Completion and refill path

The callback chain is now clear:

```text
DCL callback
  -> DCLOutputCallback (0x3add8)
  -> AM824DCLWrite::HandleDCLCallback (0x3aaf2)
  -> FillSendBuffer((completedGroup.index + 2) % numBufferGroups)
  -> FillSendBufferRawAudio, IEC958, or external-sync variant
```

`DCLOutputCallback()` obtains the group from the callback DCL argument, records
an optional timestamp, and invokes the stream's virtual completion hook.
`HandleDCLCallback()` reads the group's index at offset `0x28`, advances it by
two modulo the configured group count, and calls `FillSendBuffer()` unless the
stream is externally synchronized.

For ordinary raw-audio output, `FillSendBufferRawAudio()` at `0x380e2` is the
critical confirmation of the cadence model:

1. It selects the target buffer group by index and starts from
   `group->updateList[0]`.
2. For each packet DCL, it reads `transferBytes` from DCL offset `0x10` and
   computes the packet frame count from that fixed byte length.
3. It writes the CIP header and fills exactly that many audio-frame quadlets
   into the DCL source buffer.
4. It calls `IOFlushProcessorCache` on the filled payload buffer.

Thus the Bresenham accumulator runs **only while the DCL ring is constructed**.
The real-time refill path consumes the precomputed 5/6, 11/12, or 22/23-frame
descriptor schedule; it does not recompute or reseed the fractional cadence.
For the historical DCL program, this makes loop-period alignment a real
correctness condition rather than a presentation detail. ASFW must satisfy the
equivalent wire-level rate invariant through its own OHCI descriptor and packet
pipeline, without inheriting this callback or buffer-group design.

IDA evidence: allocator `0x35e36..0x35e9e`, update-list builder
`0x35ea0..0x35f86`, generic callback `0x3add8..0x3ae73`, stream completion
`0x3aaf2..0x3ac21`, and raw-audio refill `0x380e2..0x38419`.

## 7. Apple NuDCL 44.1 kHz blocking cadence

The later `AM824NuDCLWrite` path is not the legacy 5/6-frame scheduler. At
44.1 kHz, its data packet is always exactly 8 frames; a cycle not selected for
data is an empty packet. The rate is achieved by deciding whether a bus cycle
carries eight frames or zero frames.

### Startup state recovered from IDA

`AM824NuDCLWrite::Start()` initializes these values before it creates the
NuDCL program:

| State | Value | IDA evidence |
|---|---:|---|
| Packet phase | `0` | `0x4c0ef..0x4c0f9` |
| Next-SYT phase | `117,220,000` | `0x4c103..0x4c10d` |
| 44.1 SYT interval | `8` frames | `0x4c2a0..0x4c2b5` |
| Per-cycle phase increment | `30,720,000` | `CalculatePacketHeaderData`, `0x44d23..0x44d42` |
| Next-SYT increment | `44,582,312` | Stored at `0x10bc/0x10c0` by the 44.1 branch |
| Phase modulus | `491,520,000` | `0x44d7b`, 16 FireWire cycles |
| SYT presentation lead | `460,800,000` | `0x4c513..0x4c51d`, 15 FireWire cycles |

All phase values are FireWire ticks scaled by 10,000. One FireWire cycle is
therefore `3072 * 10000 = 30,720,000` phase units.

### Per-cycle decision

`CalculatePacketHeaderData()` does this once for each outgoing isochronous
cycle:

```cpp
packetPhase = (packetPhase + 30'720'000) % 491'520'000;

if (packetPhase has reached nextSytPhase on the 16-cycle ring) {
    sytPhase = (nextSytPhase + 460'800'000) % 491'520'000;
    nextSytPhase = (nextSytPhase + 44'582'312) % 491'520'000;
    return 8;  // data packet
}

return 0;      // no-data packet
```

The actual compare at `0x44d5a..0x44d8f` is a half-ring circular comparison,
so a stale deadline more than eight cycles behind is not mistaken for one that
is due now.

`ProcessCIPHeader()` consumes that return value:

```text
return 8: write the calculated SYT, send an 8-frame payload, advance DBC by 8
return 0: write SYT = 0xffff, send no payload, leave DBC unchanged
```

### Apple startup prefix and steady cadence

With the exact startup values above, and assuming one calculation per bus
cycle before any later resynchronization, the payload decisions begin:

```text
N N N D N D D N D D N D D D N D ...
```

`D` is eight frames and `N` is zero frames. The first three empty cycles are
a consequence of the first Apple SYT deadline being seeded 3.815 FireWire
cycles ahead of a zero packet phase. A direct simulation of the stored integer
state has 439 data packets in this first 640-cycle startup window; once this
lead is absorbed, the nominal steady cadence is 441 data packets per 640
cycles:

```text
441 data packets * 8 frames = 3528 frames per 80 ms = 44,100 frames/second
```

The 44,582,312 increment is a scaled integer approximation. It minimizes
fractional timing error; the phase machine, not a short literal `N/D` table,
is Apple's source of truth. The resynchronization path in `ProcessCIPHeader()`
can also alter the effective start phase, so captures need not begin with the
prefix above.

## 8. ASFW timing interpretation and working decision

This section records the design conclusion from the reverse-engineering work.
It is not an instruction to port DCL or NuDCL into ASFW.

Apple NuDCL's phase/deadline formulation is the behavioral reference for a
blocking 44.1-family transmitter because it keeps the whole decision in the
FireWire time domain:

```text
packet phase + next SYT deadline + presentation lead -> CIP packet decision
```

It is mathematically Bresenham-like: a fractional rate is represented by a
carried phase remainder rather than by floating point. Its advantage is not a
different rate equation. Its advantage is that the same phase state decides
all of the wire-visible timing at once: data versus no-data, SYT presentation
time, 16-cycle wrap, lead, and resynchronization.

For ASFW, the intended ownership boundary is:

| Concern | Intended ASFW owner | Rule |
|---|---|---|
| 44.1-family packet cadence, DBC, and SYT | `Audio/Wire/AMDTP` | Run in FireWire cycle/tick time. Do not derive a packet SYT from host time. |
| Bus-time to host-time relationship | Audio runtime / ZTS anchor | Map observed bus phase to CoreAudio host ticks. Do not make it the packet-cadence source. |
| OHCI descriptors and DMA completion | `Isoch/` | Carry already framed packets; remain payload-opaque. |

The relationship is one-way at steady state:

```text
FireWire cycle/SYT phase -> host-time anchor -> CoreAudio ZTS
FireWire cycle/SYT phase -> CIP/AMDTP packetizer -> OHCI transmit descriptors
```

CoreAudio ZTS needs a trustworthy sample-time-to-host-time mapping. Apple's
NuDCL reference is useful because it already treats SYT as a bus presentation
deadline, rather than as a by-product of a host callback. ASFW should preserve
that separation: host ticks explain the bus clock to CoreAudio; they do not
replace the bus clock as the source of outgoing SYT cadence.

**Working decision:** retain the Apple phase/deadline model as the reference
for ASFW blocking 44.1-family behavior. Keep the original concepts of packet
phase, next-SYT phase, presentation lead, circular comparison, and explicit
resynchronization. Re-express them with ASFW's own DriverKit ownership and
integer types; do not copy Apple DCL/NuDCL APIs, memory layout, or callbacks.

A generic rational/Bresenham helper can prove the average rate, but it is not
a replacement for the Apple model by itself. It does not determine the Apple
startup seed, SYT lead, wrap convention, or resynchronization semantics.

## 9. Saffire (DICE) phase recovery comparison

The Saffire kext shows a different, and more useful, way to handle a real
44.1-family clock. It does not keep generating a nominal fractional 44.1
phase step after startup. It measures the clock expressed by received SYT
values, delays that measurement safely, and replays it into the output SYT
phase. The result follows the device's actual oscillator, including its
fractional pattern and drift.

### Relevant Saffire functions

| IDA address | Function | Finding |
|---:|---|---|
| `0x14052` | `GetSYTIntervalBySampleRate(uint)` | Returns the standard fixed interval: 44.1/48 kHz -> `8`, 88.2/96 kHz -> `16`, 176.4/192 kHz -> `32`. |
| `0x141e4` | `SampleRateToSYTDifference(uint)` | Returns a bootstrap integer offset per frame: 44.1 kHz -> `557`; `557 * 8 = 4456` offset ticks per SYT interval. |
| `0x1463a` | `SYTDiffInOffsets(ushort, ushort)` | Converts two SYT values to `(cycle * 3072 + offset)` and obtains their wrapped difference. |
| `0xcf24` | `Saffire::ReadFirewireBuffers(ISOCH_GROUP_INFO *)` | Learns the incoming SYT-delta sequence into a ring. |
| `0xc9c2` | `Saffire::adjustOutputPhase(_DICE_STREAM_STRUCT *, uint, uint)` | Wrap-aware, deadbanded output-phase correction. |
| `0xe778` | `Saffire::FillFirewireBuffers(ISOCH_GROUP_INFO *)` | Replays learned deltas to generate outgoing SYT values. |

The nominal 44.1 interval in offset ticks is:

```text
24,576,000 * 8 / 44,100 = 4,458.231... offset ticks
```

Thus the `4456` value from `557 * 8` is only a practical startup fallback;
it is neither exact nor the locked cadence source.

For an ideal nominal source, the observed interval sequence is `4458` or
`4459` ticks, with 34 long intervals in every 147 SYT events. A real device
clock can move that sequence; preserving the measurements is therefore more
valuable than regenerating this nominal pattern.

### Measured-delta ring

For each valid received SYT, `ReadFirewireBuffers()` derives an exact observed
interval. The first interval uses the `4456` bootstrap because no predecessor
SYT exists yet; subsequent intervals are calculated from adjacent received
timestamps. The kext writes those values to a 512-entry `uint16_t` ring.

The writer begins 256 entries ahead of the reader. On every update it replaces
one old value in the 256-sample running window and calculates:

```text
delta[write] = measuredDelta
write = (write + 1) & 511

runningDeltaSum = runningDeltaSum - delta[read] + measuredDelta
read = (read + 1) & 511
meanDelta = runningDeltaSum >> 8       // divide by 256
```

After its 512-interval warm-up threshold is reached, it enables output
synchronization. The 512-entry size lets output consume a sequence 256 entries
behind the current input writer: it is a deliberate measured-phase delay, not
a mere averaging buffer.

`FillFirewireBuffers()` initializes each output stream's delta cursor to
`(write + 256) & 511`. Once synchronized, it advances the scheduled output
phase by the actual stored value and encodes that phase as SYT:

```text
if (!synchronized)
    emitNoSyt()
else {
    outputSyt = encodeSyt(outputPhase)
    outputPhase = (outputPhase + delta[outputCursor]) % phaseModulus
    outputCursor = (outputCursor + 1) & 511
}
```

The phase modulus is `0x0bb80000` offset units, or `64,000 * 3072`. The
comparison helper `extOffsetDiff()` chooses the signed shortest difference in
that circular domain, so all phase decisions remain valid across its wrap.

### Output-phase servo

The delta ring handles rate and the 44.1 fractional pattern. A separate
controller handles placement of that sequence in time. At the start of a
buffer-fill batch Saffire compares the candidate output phase with the latest
observed input timestamp. It converts that wrap-aware error through the
256-interval running sum, tracks the signed residual, and only moves the
output deadline when a forced-correction flag is set or the residual exceeds
a per-stream deadband. The correction is forward in the phase ring, preserving
monotonic scheduled presentation times.

In fresh pseudocode, deliberately independent of the kext's layout:

```text
error = shortestCircularDifference(candidatePhase, observedInputPhase)
forwardCorrection, signedResidual = deriveForwardCorrection(
    error, framesPerSyt * 256, runningDeltaSum)

if (forceCorrection || abs(signedResidual) > correctionDeadband)
    candidatePhase = (candidatePhase + forwardCorrection) % phaseModulus

return candidatePhase
```

This controller is not a second sample-rate accumulator. It is phase
recovery: measured SYT deltas define cadence; the servo keeps that cadence at
the desired bus-time lead despite queue delay, timestamp movement, and drift.

### Apple NuDCL versus Saffire

| Concern | Apple NuDCL | Saffire |
|---|---|---|
| 44.1 cadence source | Synthesized fixed-point next-SYT deadline (`44,582,312` scaled units). | Replayed received-SYT delta sequence, after a short integer bootstrap. |
| Fractional 44.1 error | Carried in Apple-owned packet/deadline phase. | Present directly in the measured delta sequence, e.g. alternating nearby integer offset deltas. |
| Clock drift | Handled through Apple resynchronization around its phase model. | Naturally appears in the measured ring; placement is additionally corrected by the phase servo. |
| Output timing domain | 16-cycle scaled phase ring with explicit SYT lead. | Extended offset phase ring with wrap-aware correction and queue lead. |
| Architectural lesson for ASFW | The packetizer must own bus-time cadence and SYT presentation. | A received-SYT-driven recovery source can be an even better clock input when the device and stream topology provide one. |

The important distinction is not “Bresenham versus no Bresenham.” Apple
generates the rational cadence from known rate and bus phase; Saffire learns
the already-realized cadence from the device clock and preserves it. For an
ASFW stream locked to device-received SYT, the Saffire approach explains why a
timestamp-driven input clock is preferable to nominal rate arithmetic alone.
It does not change the ASFW layering decision: this belongs to the
`Audio/Wire` cadence/recovery model and its neutral timing inputs, never to
payload-opaque `Isoch/`.

## 10. FFADO 2.4.9 comparison (secondary)

FFADO is a useful comparison because version 2.4.9 contains both a generic
IEC 61883 rational packetizer and the timestamp-driven AMDTP processor used by
the generic streaming path. These are different algorithms. The latter is the
relevant one for FFADO's actual generic AM824 transmission.

This section is derived from the supplied read-only GPL source tree at
`/Users/mrmidi/DEV/FirWireDriver/OTHER/libffado-2.4.9`. It records behavior
only; none of its code belongs in ASFW.

### 10.1 Generic CIP helper: exact rational cadence

`src/libstreaming/util/cip.c` has an explicit integer-plus-remainder fraction
type. `iec61883_cip_init()` sets its default mode to `BLOCKING_EMPTY`, creates
`samples_per_cycle = rate / 8000`, and independently creates the SYT phase
increment `ticks_per_syt_offset = 24,576,000 * syt_interval / rate`
([`cip.c:81-107`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libstreaming/util/cip.c)).

At 44.1 kHz this is the conventional rational representation:

```text
samplesPerCycle = 44,100 / 8,000 = 5 + 41/80
ticksPerSyt     = 24,576,000 * 8 / 44,100 = 4,458 + 34/147
```

For `BLOCKING_EMPTY`, the header helper adds `samplesPerCycle` to a carried
`readySamples` fraction every FireWire cycle. It emits either a full
`sytInterval` block or zero events, then subtracts the emitted event count
from the same fraction ([`cip.c:131-158`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libstreaming/util/cip.c)).
Fresh pseudocode for the 44.1 kHz case is:

```text
ready += 5 + 41/80

if (floor(ready) >= 8)
    events = 8
else
    events = 0

ready -= events
dbc += events
```

From a zero seed this gives the familiar blocking prefix
`N, D, D, N, D, D, ...`, where `D` is eight frames. It has 441 data packets in
640 FireWire cycles, exactly like the rate equation behind Apple's NuDCL
steady cadence.

The same helper carries the SYT offset as a second exact fraction. When DBC
places the SYT-bearing frame in a data packet, it writes the current
`cycleCount/offset` and advances it by `ticksPerSyt`; otherwise it writes
`0xffff` ([`cip.c:160-176`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libstreaming/util/cip.c)).
This is a clean two-accumulator formulation: one accumulator selects 8/0
payload cadence, and another advances the fractional SYT offset.

### 10.2 Active generic AMDTP path: SYT-fed timestamp graph

The generic `AmdtpTransmitStreamProcessor` calls `iec61883_cip_init()` during
preparation ([`AmdtpTransmitStreamProcessor.cpp:449-469`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libstreaming/amdtp/AmdtpTransmitStreamProcessor.cpp)),
but its packet-producing methods do not call `iec61883_cip_fill_header()`.
Instead, they use their own timestamped-buffer scheduling path. In this source
revision, `m_cip_status` is initialized but is not consumed by the generic
AMDTP packet-generation methods.

The active path is:

```text
received SYT -> full FireWire tick timestamp -> TimestampedBuffer DLL
             -> SyncSource ticks-per-frame -> transmit buffer timestamps
             -> output presentation deadline -> SYT in transmitted packet
```

1. The receive processor accepts a valid SYT, reconstructs it as full FireWire
   ticks, and stamps its received frames with that value
   ([`AmdtpReceiveStreamProcessor.cpp:97-122`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libstreaming/amdtp/AmdtpReceiveStreamProcessor.cpp),
   [`:175-235`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libstreaming/amdtp/AmdtpReceiveStreamProcessor.cpp)).
   44.1 kHz uses `syt_interval = 8` in both directions
   ([`AmdtpReceiveStreamProcessor.cpp:54-70`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libstreaming/amdtp/AmdtpReceiveStreamProcessor.cpp),
   [`AmdtpTransmitStreamProcessor.cpp:415-430`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libstreaming/amdtp/AmdtpTransmitStreamProcessor.cpp)).
2. `TimestampedBuffer` compares each observed timestamp with the predicted
   timestamp, updates a delay-locked loop, and exposes the resulting measured
   ticks-per-frame rate ([`TimestampedBuffer.cpp:1025-1102`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libutil/TimestampedBuffer.cpp)).
3. The stream manager uses that sync-source rate to assign a future
   presentation timestamp to every transmit period. The future offset is the
   configured transmit-ring depth in frames ([`StreamProcessorManager.cpp:1488-1511`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libstreaming/StreamProcessorManager.cpp)).
4. The transmit processor reads the head presentation timestamp, subtracts its
   transfer delay, and sends the fixed eight-frame packet only when it is in
   the allowed FireWire-cycle window. If it is early, it emits an empty
   no-data packet instead ([`AmdtpTransmitStreamProcessor.cpp:100-260`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libstreaming/amdtp/AmdtpTransmitStreamProcessor.cpp)).
   A data packet encodes `TICKS_TO_SYT(presentationTime)` and increments DBC
   by eight; a no-data packet uses `FDF_NODATA`, `SYT = 0xffff`, and leaves DBC
   unchanged ([`AmdtpTransmitStreamProcessor.cpp:372-410`](../../FirWireDriver/OTHER/libffado-2.4.9/src/libstreaming/amdtp/AmdtpTransmitStreamProcessor.cpp)).

The DLL update, expressed independently of FFADO's code, is:

```text
error = shortestWrappedDifference(observedPacketTimestamp, predictedNextTimestamp)

tailTimestamp = predictedNextTimestamp
predictedNextTimestamp += phaseGain * error + periodTicks
periodTicks += rateGain * error
measuredTicksPerFrame = periodTicks / updateFrames
```

This is phase/rate recovery from received SYT, but it occurs at the
timestamped-buffer and stream-manager layer rather than in a per-cycle packet
phase machine. The transmit packetizer consumes already timestamped audio
blocks and uses their presentation deadline as its authority.

### 10.3 Comparison with Apple and Saffire

| Concern | Apple NuDCL | Saffire | FFADO active AMDTP |
|---|---|---|---|
| Nominal 44.1 fallback | Fixed-point packet and next-SYT phase. | `557 * 8` tick bootstrap. | Nominal ticks-per-frame initializes `TimestampedBuffer`. |
| Locked cadence source | Apple phase/deadline plus resynchronization. | Measured received-SYT delta replay. | Received SYT feeds a DLL; the SyncSource rate timestamps future transmit blocks. |
| Output packet decision | Per FireWire cycle: eight frames or no data. | Replayed output phase and queue lead. | Send a fixed eight-frame block when its presentation timestamp reaches the transmit window; otherwise no-data. |
| SYT construction | Explicit scaled phase and presentation lead. | Encoded from measured/replayed output phase. | `TICKS_TO_SYT` of the buffered block's presentation timestamp. |
| Main limitation | Binary recovery policy remains incomplete. | Requires an eligible incoming SYT stream. | Uses floating-point DLL state and a Linux user-space scheduling graph; its exact timing behavior is not an ASFW packetizer blueprint. |

FFADO therefore supports two separate conclusions. Its generic CIP helper
confirms the normal rational 8/0 and fractional-SYT mathematics. Its active
AMDTP path supports the stronger architectural idea that received SYT should
discipline a sample-time-to-bus-time clock, and transmit packets should carry
presentation timestamps from that clock. The latter is conceptually close to
the ASFW `HostClockAnchor`/ZTS relationship, but ASFW must keep the
wire-visible cadence and SYT emission in `Audio/Wire` with integer FireWire
time, not port FFADO's floating-point user-space scheduler.
